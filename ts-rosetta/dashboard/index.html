高頻度ライブ盤面 — リアクティビティ比較（2026-07-10 実測）

各アプリの #board：同一ティックストリーム（毎秒 N 回、銘柄の10%をランダムウォーク更新）を5通りの配線で描画。

work/s（1,000銘柄 @60tps・touched ≈ 5,900/s）実測意味
React naive（useState 配列を毎tick差替）62,161= 行数×tick数。全行が毎tick再レンダー＝触られた行の10.6倍
React optimized（外部ストア+行単位購読）5,8724つの異なる機構が「変わった行だけ」に収束（work ≈ touched）
Vue（reactive オブジェクト in-place 変更）5,887
Angular（signal/行 + OnPush + zone外実行）5,887
Solid（store パス更新 → セル単位 effect）5,887

一方 fps はどの負荷でもフレームワーク間でほぼ差がつかない（CPU 4x絞りで全員23-34、5,000行で全員19-29）— 崩壊時の律速は巨大テーブルのブラウザ layout（共通コスト）。教訓: ①リアクティビティ戦略の差は fps でなく CPU 予算の消費に出る ②fps を守る本命は仮想化 ③Solid のバンドルは 19.3kB と React の約1/10。詳細と罠→対策コードは docs/reactivity.md。再計測は node tools/verify-board.mjs。
