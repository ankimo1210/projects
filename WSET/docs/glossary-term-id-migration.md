## 決定の記録方法

各候補の`decision`は次のいずれかとする。

- `pending_expert_review`: 未決定。`canonicalTermID`は`null`のままにする。
- `merge`: 同一カードへ統合する。既存2 IDの一方だけを`canonicalTermID`に選ぶ。理由、レビュー担当者、`YYYY-MM-DD`のレビュー日を必須とする。
- `keep_separate`: 意図的に別カードを維持する。理由、レビュー担当者、レビュー日を必須とし、`canonicalTermID`は`null`にする。

新しい第3のIDは作らない。既存IDの一方を残すことで、少なくともそのIDの端末履歴と外部参照をそのまま維持する。

## `merge`決定後の正本更新

1. 残す行の概要・詳細・カテゴリ・表示名を専門家が確定する。
2. 退役IDの`公開`を`N`にし、正規IDは`Y`を維持する。生成器は、承認済み退役ID以外の非公開行を拒否する。
3. `関連用語IDs`と格付け項目、記述式、産地マップ、無料枠に旧IDがあれば正規IDへ更新する。各生成器・validatorの参照切れ検査を通す。
4. 退役カードの日本語・英語・フランス語名と既存別名は、生成時に正規カードの別名へ引き継ぐ。これにより検索と関連問題の逆引きを維持する。
5. 無料枠で両IDが選ばれているグレラ組を統合する場合、重複分を別の公開用語へ置き換え、無料60語・カテゴリ分散契約を維持する。
6. `python3 scripts/build_reference_pack.py`と`make verify`を実行し、生成物の`termIDMigrations`が承認内容だけを含むことを確認する。

## SwiftData・SRSの合流規則

アプリ起動時、生成パックの承認済み`termIDMigrations`を一度適用する。

- 旧IDだけに履歴がある場合は、同じレコードの`termID`を正規IDへ変更する。
- 旧IDと正規IDの両方に履歴がある場合は、ブックマークを論理和、閲覧回数・復習回数・正答回数を加算する。
- 最終閲覧日は新しい方を採用する。
- 次回期限、間隔、最終正誤は、`lastReviewedAt`が新しい方を採用する。
- 合流後は旧IDレコードを削除する。旧IDがなくなるため再起動しても再加算されず、処理は冪等になる。

## バックアップ互換性

schemaVersion 1の旧バックアップは引き続き復元できる。

- 復元前に端末内の旧ID履歴を同じ規則で合流する。
- バックアップ内に旧IDと正規IDが共存する場合、異なる元IDの回数を加算し、最新のSRS期限を採用して正規ID1件へまとめる。
- 同じバックアップを再度復元しても、端末側との比較は従来どおり最新スナップショット優先とし、回数を重複加算しない。
- 新規バックアップは正規IDだけを書き出す。

購入権利や秘密情報は引き続きバックアップ対象外であり、この変更だけではバックアップschemaVersionを上げない。
