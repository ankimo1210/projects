# RealtimeTicker — Design

Date: 2026-07-15
Status: Approved
Project: CsharpApp (WPF, net9.0-windows, zero runtime dependencies)

## Purpose

A learning-oriented sample app that exercises the core of WPF in one place:
XAML layout, data binding, MVVM, `INotifyPropertyChanged`,
`ObservableCollection`, `ICommand`, value converters, `DispatcherTimer`-driven
live updates, and custom drawing with `Canvas` + `Polyline`.

The app shows a live-updating price ticker for 5 hard-coded instruments.
Selecting a row shows a live line chart of that instrument. Prices are
self-generated by a GBM random walk — fully offline, no APIs, no runtime
NuGet dependencies.

## Success criteria

- `dotnet.exe run --project CsharpApp.csproj` opens a window with the first
  instrument selected and ticking automatically; the grid recolors live.
- Selecting any row immediately shows that instrument's retained recent
  history; the chart continues updating and follows resize.
- Start/pause works; the update interval is adjustable while running via a
  slider from 100 ms to 1,000 ms in 50 ms steps.
- Model layer has no `System.Windows` reference and is covered by xUnit tests.
- `dotnet.exe test CsharpApp.Tests/CsharpApp.Tests.csproj` discovers the tests
  named in this document and all discovered tests pass. Zero discovered tests
  is a failure even if the command exits with code 0.

## Architecture (MVVM, 3 layers)

```
CsharpApp/
├── Models/
│   ├── Instrument.cs        # symbol, price, previous price, history buffer
│   ├── PriceHistoryBuffer.cs # fixed-capacity, observable ring buffer
│   └── GbmPriceEngine.cs    # GBM random-walk generator (pure C#, UI-free)
├── ViewModels/
│   ├── ViewModelBase.cs     # INotifyPropertyChanged implementation
│   ├── RelayCommand.cs      # ICommand implementation (hand-rolled)
│   ├── PriceDirection.cs    # Up, Down, Unchanged UI-state enum
│   ├── TickerViewModel.cs   # one instrument row (price, change %, direction)
│   └── MainViewModel.cs     # instrument list, selection, start/stop, timer
├── Views/
│   ├── SparklineControl.cs  # Canvas+Polyline custom chart control
│   └── PriceFlashBehavior.cs # view-only row animation behavior
├── Converters/
│   └── PriceDirectionToBrushConverter.cs  # direction → display brush
├── MainWindow.xaml          # left: ticker DataGrid, right: chart pane
├── MainWindow.xaml.cs       # DataContext and window lifecycle only
└── CsharpApp.Tests/
    ├── CsharpApp.Tests.csproj # net9.0-windows; references the app project
    ├── Models/               # model xUnit tests
    ├── ViewModels/           # command/timer lifecycle xUnit tests
    └── Views/                # XAML binding and chart behavior tests
```

Design rules:

- Model layer must not reference `System.Windows` → unit-testable.
- `Instrument` is the single source of truth for current price, previous price,
  and history. `TickerViewModel` wraps it and raises binding notifications; it
  must not keep an independent price copy.
- `MainViewModel.Instruments` is an `ObservableCollection<TickerViewModel>`;
  its five entries are created once at startup and are not added or removed by
  the UI.
- `PriceHistoryBuffer` has capacity 300, implements `IReadOnlyList<double>` and
  `INotifyCollectionChanged`, and remains UI-framework-free. An append emits an
  add notification while below capacity; replacing the oldest point at capacity
  emits a reset notification.
- Randomness is injected as a `Random` instance. Tests create two engines with
  separate `new Random(sameSeed)` instances and compare their generated series;
  tests do not assert runtime-specific hard-coded `Random` outputs.
- The application project has no third-party runtime NuGet packages. The test
  project may reference xUnit plus the test SDK/adapter packages required for
  `dotnet test`; all such packages must remain test-only.

## Data flow

```
DispatcherTimer (default 250 ms, fires on UI thread)
  → MainViewModel.Tick()
  → Δt ← UpdateIntervalMs / 1000.0 (simulated seconds)
  → GbmPriceEngine.NextPrice() for every instrument
       S ← S · exp((μ − σ²/2)Δt + σ√Δt · Z)
  → Instrument updates previous/current price and appends to its history
  → TickerViewModel raises property updates
  → INotifyPropertyChanged → DataGrid rows update (green/red flash)
  → every history emits a collection change
  → SparklineControl receives and redraws only the selected history
```

`μ` is drift per simulated second and `σ` is volatility per square root of a
simulated second. Because `Δt` follows the configured timer interval, changing
the slider changes sampling density but keeps the approximate distribution of
movement per wall-clock second stable. Timer scheduling delay is deliberately
ignored: this sample uses the configured interval, not measured elapsed time.

The five offline dummy instruments are fixed so manual runs and screenshots have
stable labels and comparable behavior:

| Symbol | Initial price | μ / second | σ / √second |
|---|---:|---:|---:|
| ALPHA | 100.00 | 0.0005 | 0.008 |
| BRAVO | 250.00 | -0.0002 | 0.012 |
| CHARLIE | 75.00 | 0.0010 | 0.018 |
| DELTA | 40.00 | 0.0000 | 0.025 |
| NOVA-X | 1,000.00 | 0.0015 | 0.040 |

Every instrument starts with its initial price already in its history and is
updated on every tick, whether selected or not. Selection changes only which
history the chart displays. Pause freezes prices and all histories.

`GbmPriceEngine` generates `Z` with the Box–Muller transform. It uses
`u1 = 1.0 - Random.NextDouble()` so the logarithm never receives zero, and it
does not cache the second normal variate; one price step always consumes exactly
two random values.

## UI layout

- **Left pane**: `DataGrid` — symbol / last price / change % / direction
  arrow. Text turns green/red by direction; row background flashes on update.
- **Right pane**: line chart of the selected instrument (custom
  `Canvas`+`Polyline`, resize-aware) plus a large last-price readout.
- **Top bar**: start ▶ / pause ⏸ buttons (`RelayCommand`), update-interval
  slider (100 ms – 1,000 ms, 50 ms ticks, snap-to-tick enabled).

Initial and command behavior:

- The first instrument is selected and the timer starts at 250 ms when the
  window opens.
- Start is executable only while paused; Pause is executable only while running.
  Both commands raise `CanExecuteChanged` when `IsRunning` changes.
- Changing `UpdateIntervalMs` immediately updates the running timer's interval.
- Closing the window stops the timer and detaches its tick handler.

Price presentation rules:

- Change percent is the unrounded tick-to-tick change:
  `(Price - PreviousPrice) / PreviousPrice * 100`.
- Direction is `Up`, `Down`, or `Unchanged`, based on unrounded prices. The
  initial row is unchanged. Arrows are `▲`, `▼`, and `–`; foreground colors are
  green, red, and neutral gray respectively.
- Price is displayed with two decimals and change with an explicit sign and two
  decimals. Formatting does not affect direction.
- Each non-zero update restarts a 150 ms row-background animation from a light
  direction color to transparent. This is a visual concern and may be handled
  in the view or a view-only behavior; it does not put brushes in the model or
  view model. Unchanged updates do not flash.

`SparklineControl` contract:

- Expose a `Values` dependency property typed as `IReadOnlyList<double>`.
- When `Values` changes, unsubscribe from the old source and subscribe to the
  new source when it implements `INotifyCollectionChanged`.
- Redraw on collection change, dependency-property replacement, and
  `SizeChanged`. All callbacks run on the UI thread.
- Zero points clears the canvas; one point is drawn as a centered dot; an
  all-equal series is drawn as a horizontal line at vertical center.
- Otherwise, point indices span the available width and finite min/max values
  span the height with 8 px padding. Non-positive control dimensions defer
  drawing until the next size change.

## Error handling and numeric constraints

There is no external I/O (no network, file, or API access). The remaining
constraints are UI-thread affinity, parameter validation, floating-point range,
and degenerate chart geometry.

- `DispatcherTimer` fires on the UI thread, so no `Dispatcher.Invoke` is needed.
  This is documented in code comments as a learning point.
- `GbmPriceEngine.NextPrice` rejects a non-finite or non-positive current price,
  a non-finite `μ`, a non-finite or negative `σ`, and a non-finite or
  non-positive `Δt` with `ArgumentOutOfRangeException`.
- Although mathematical GBM is positive, `Math.Exp` can underflow to zero or
  overflow to infinity. If a computed candidate is non-finite or non-positive,
  the engine returns the last valid current price and never commits an invalid
  value to an instrument.
- `MainViewModel` validates `UpdateIntervalMs` as 100–1,000 before assigning
  `DispatcherTimer.Interval`; programmatic out-of-range values are rejected.
- Chart drawing handles empty, single-point, flat, and temporarily zero-sized
  inputs according to the rules above.

## Testing

- New `CsharpApp.Tests` project (xUnit), covering the Model layer plus
  deterministic ViewModel command/timer behavior:
  - GBM: two engines with the same seed reproduce the same series.
  - GBM: configured instruments remain finite and positive over 10,000 steps.
  - GBM: `μ = 0`, `σ = 0` leaves the price exactly unchanged.
  - GBM: invalid inputs throw and a non-finite computed candidate falls back to
    the last valid price.
  - History: the initial price is present, insertion order is preserved, count
    never exceeds 300, and the oldest point is dropped on the 301st append.
  - Main ViewModel: starts with five instruments and the first selected, updates
    every history, pauses without mutation, resumes, enforces interval bounds,
    and exposes the correct command states. This runs on an STA dispatcher.
  - Views: XAML loads with the expected bindings and slider configuration; the
    sparkline handles empty, single-point, flat, live, and resized histories;
    repeated same-direction updates restart the row flash while unchanged prices
    do not flash. These checks also run on an STA dispatcher.
- Tests are run explicitly with
  `dotnet.exe test CsharpApp.Tests/CsharpApp.Tests.csproj`. The output must list
  the tests above and report a non-zero total; command exit code alone is not
  sufficient evidence.
- ViewModels and drawing are verified manually with
  `dotnet.exe run --project CsharpApp.csproj`:
  1. The first row is selected and all five prices begin updating at 250 ms.
  2. Pause freezes every price and history; Start resumes them.
  3. Moving the slider changes cadence immediately at both limits.
  4. Switching rows immediately shows retained history and continued updates.
  5. Repeated same-direction moves restart the 150 ms flash; unchanged is gray
     and does not flash.
  6. The chart remains valid while resizing and for one-point or flat history.
  7. Closing the window produces no timer callbacks or unhandled exceptions.
- UI-automation testing is out of scope.

## Out of scope (YAGNI)

Real market-data feeds, add/remove-instrument UI, multiple simultaneous
charts, theming, settings persistence.
